                              TABLE STRUCTURES
================================================================================

--------------------------------------------------------------------------------
1. ADMINS TABLE
--------------------------------------------------------------------------------
Purpose: Stores administrator accounts for system management

Columns:
  - id              SERIAL PRIMARY KEY      (Auto-increment ID)
  - username        VARCHAR(100) UNIQUE     (Login username)
  - email           VARCHAR(255) UNIQUE     (Administrator email)
  - password_hash   VARCHAR(255)            (BCrypt hashed password)
  - full_name       VARCHAR(255)            (Full name of admin)
  - status          VARCHAR(50) DEFAULT 'active'  (Account status)
  - created_at     TIMESTAMP DEFAULT NOW   (Creation date)
  - updated_at     TIMESTAMP DEFAULT NOW   (Last update)
  - last_login     TIMESTAMP               (Last login time)

Indexes:
  - Primary key on id
  - Unique on username
  - Unique on email

Default Accounts:
  - Username: admin | Email: admin@sentrina.com | Password: password (hashed)

--------------------------------------------------------------------------------
2. STAFFS TABLE
--------------------------------------------------------------------------------
Purpose: Stores staff accounts for operations

Columns:
  - id              SERIAL PRIMARY KEY      (Auto-increment ID)
  - username        VARCHAR(100) UNIQUE   (Login username)
  - email            VARCHAR(255) UNIQUE   (Staff email)
  - password_hash   VARCHAR(255)            (BCrypt hashed password)
  - full_name       VARCHAR(255)            (Full name of staff)
  - position        VARCHAR(100) DEFAULT 'staff'  (Job position)
  - status          VARCHAR(50) DEFAULT 'active'  (Account status)
  - created_at     TIMESTAMP DEFAULT NOW   (Creation date)
  - updated_at     TIMESTAMP DEFAULT NOW   (Last update)
  - last_login     TIMESTAMP               (Last login time)

Indexes:
  - Primary key on id
  - Unique on username
  - Unique on email

Default Accounts:
  - Username: staff | Email: staff@sentrina.com | Position: General Staff

--------------------------------------------------------------------------------
3. RESIDENTS TABLE (Homeowners/Users)
--------------------------------------------------------------------------------
Purpose: Stores homeowner/resident information

Columns:
  - id              SERIAL PRIMARY KEY      (Auto-increment ID)
  - username        VARCHAR(100) UNIQUE     (Login username)
  - password_hash   VARCHAR(255)            (BCrypt hashed password)
  - full_name       VARCHAR(255)            (Full name of resident)
  - lot_number      VARCHAR(50)             (Unit/Lot number)
  - block           VARCHAR(50)             (Block/Tower)
  - phase           VARCHAR(50)             (Phase/Section)
  - email           VARCHAR(255)            (Email address)
  - phone           VARCHAR(50)             (Contact number)
  - profile_image  TEXT                     (Profile image URL)
  - role            VARCHAR(50) DEFAULT 'homeowner'  (User role)
  - status          VARCHAR(50) DEFAULT 'active'  (Account status)
  - date_registered DATE DEFAULT CURRENT_DATE  (Registration date)
  - last_login     TIMESTAMP               (Last login time)
  - created_at     TIMESTAMP DEFAULT NOW   (Creation date)
  - updated_at     TIMESTAMP DEFAULT NOW   (Last update)

Indexes:
  - Primary key on id
  - Unique on username
  - idx_residents_lot_number
  - idx_residents_block
  - idx_residents_status
  - idx_residents_username

Sample Data (4 residents):
  - Juan dela Cruz | Lot: 101 | Block: A | Email: juan@example.com
  - Maria Santos | Lot: 102 | Block: A | Email: maria@example.com
  - Pedro Garcia | Lot: 201 | Block: B | Email: pedro@example.com
  - Ana Reyes | Lot: 202 | Block: B | Email: ana@example.com

--------------------------------------------------------------------------------
4. VIOLATIONS TABLE
--------------------------------------------------------------------------------
Purpose: Track resident violations and penalties

Columns:
  - id              SERIAL PRIMARY KEY      (Auto-increment ID)
  - lot_number      VARCHAR(50)             (Unit/Lot number)
  - resident_name  VARCHAR(255)            (Name of resident)
  - violation_type VARCHAR(100)             (Type of violation)
  - description    TEXT                     (Violation details)
  - date_issued    DATE DEFAULT CURRENT_DATE  (Date issued)
  - status         VARCHAR(50) DEFAULT 'pending'  (Status: pending/settled)
  - penalty        DECIMAL(10,2)             (Penalty amount in PHP)
  - created_at     TIMESTAMP DEFAULT NOW   (Creation date)
  - updated_at     TIMESTAMP DEFAULT NOW   (Last update)

Indexes:
  - Primary key on id
  - idx_violations_lot_number
  - idx_violations_status
  - idx_violations_date_issued

Status Values:
  - pending = Not yet settled
  - settled = Violation resolved

Violation Types:
  - Noise Violation
  - Illegal Parking
  - Garbage Violation
  - (Others as defined by management)

Sample Data:
  - Lot 101 - Juan dela Cruz - Noise Violation - Loud music after 10 PM - 500 PHP - pending
  - Lot 102 - Maria Santos - Illegal Parking - Parked in fire exit - 300 PHP - settled
  - Lot 201 - Pedro Garcia - Garbage Violation - Improper waste disposal - 200 PHP - pending

--------------------------------------------------------------------------------
5. BILLS TABLE
--------------------------------------------------------------------------------
Purpose: Store billing statements for residents

Columns:
  - id              SERIAL PRIMARY KEY      (Auto-increment ID)
  - lot_number     VARCHAR(50)             (Unit/Lot number)
  - resident_name  VARCHAR(255)            (Name of resident)
  - bill_type      VARCHAR(100)            (Type of bill)
  - bill_reference VARCHAR(100)            (Bill reference number)
  - billing_period VARCHAR(100)            (Billing period)
  - amount         DECIMAL(10,2) NOT NULL  (Bill amount in PHP)
  - due_date       DATE                   (Payment due date)
  - status         VARCHAR(50) DEFAULT 'unpaid'  (Status: unpaid/paid)
  - date_paid      DATE                    (Date paid)
  - payment_method VARCHAR(50)             (Method: GCash/Cash/etc)
  - amount_paid   DECIMAL(10,2)            (Amount paid)
  - created_at     TIMESTAMP DEFAULT NOW   (Creation date)
  - updated_at     TIMESTAMP DEFAULT NOW   (Last update)

Indexes:
  - Primary key on id
  - idx_bills_lot_number
  - idx_bills_status
  - idx_bills_due_date

Status Values:
  - unpaid = Not yet paid
  - paid = Fully paid

Bill Types (examples):
  - Monthly Dues
  - Association Fee
  - Parking Fee
  - Utilities

Reference Format: BILL-{lot_number}-{sequence}

--------------------------------------------------------------------------------
6. PAYMENTS TABLE
--------------------------------------------------------------------------------
Purpose: Track resident payments

Columns:
  - id              SERIAL PRIMARY KEY      (Auto-increment ID)
  - lot_number     VARCHAR(50)             (Unit/Lot number)
  - resident_name  VARCHAR(255)            (Name of resident)
  - bill_reference VARCHAR(100)            (Reference to bill)
  - amount         DECIMAL(10,2) NOT NULL  (Payment amount in PHP)
  - payment_date   DATE DEFAULT CURRENT_DATE  (Date of payment)
  - payment_method VARCHAR(50)             (Method: GCash/Cash/Bank Transfer)
  - status         VARCHAR(50) DEFAULT 'pending'  (Status: pending/approved/rejected)
  - approved_date  DATE                    (Date approved)
  - created_at     TIMESTAMP DEFAULT NOW   (Creation date)
  - updated_at     TIMESTAMP DEFAULT NOW   (Last update)

Indexes:
  - Primary key on id
  - idx_payments_lot_number
  - idx_payments_status
  - idx_payments_payment_date

Status Values:
  - pending = Awaiting approval
  - approved = Payment confirmed
  - rejected = Payment denied

Payment Methods:
  - GCash
  - Cash
  - Bank Transfer
  - Credit Card

Sample Data:
  - Lot 102 - Maria Santos - BILL-102-001 - 1500 PHP - GCash - approved
  - Lot 101 - Juan dela Cruz - BILL-101-001 - 1500 PHP - Cash - pending

--------------------------------------------------------------------------------
7. AUDIT_LOGS TABLE
--------------------------------------------------------------------------------
Purpose: Track all user activities for security and compliance

Columns:
  - id              SERIAL PRIMARY KEY      (Auto-increment ID)
  - user_name       VARCHAR(255)            (Name of user)
  - user_role       VARCHAR(50)             (Role: admin/staff/homeowner)
  - action          VARCHAR(100)             (Action performed)
  - module          VARCHAR(100)            (Module affected)
  - description    TEXT                     (Detailed description)
  - ip_address      VARCHAR(50)               (User's IP address)
  - profile_image  TEXT                     (User's profile image)
  - status         VARCHAR(50) DEFAULT 'success'  (Status of action)
  - timestamp      TIMESTAMP DEFAULT NOW   (Time of action)

Indexes:
  - Primary key on id
  - idx_audit_logs_user_name
  - idx_audit_logs_user_role
  - idx_audit_logs_timestamp

User Roles:
  - admin = Administrator
  - staff = Staff Member
  - homeowner = Resident/Homeowner

Actions (examples):
  - Login
  - Logout
  - Create
  - Update
  - Delete
  - View

Modules (examples):
  - Authentication
  - Residents
  - Billing
  - Violations
  - Payments

Sample Data:
  - Peter Admin | admin | Login | Authentication | Admin logged in | 127.0.0.1 | success
  - John Staff | staff | View | Residents | Staff viewed residents list | 127.0.0.1 | success
  - Peter Admin | admin | Create | Billing | Admin created new bill | 127.0.0.1 | success

--------------------------------------------------------------------------------
8. NOTIFICATIONS TABLE
--------------------------------------------------------------------------------
Purpose: Store user notifications and alerts

Columns:
  - id              SERIAL PRIMARY KEY      (Auto-increment ID)
  - user_id         INTEGER REFERENCES users(id)  (Reference to user)
  - title           VARCHAR(255)            (Notification title)
  - message         TEXT                     (Notification message)
  - type            VARCHAR(50) DEFAULT 'info'  (Type: info/warning/error)
  - is_read         BOOLEAN DEFAULT FALSE   (Read status)
  - created_at      TIMESTAMP DEFAULT NOW   (Creation date)

Indexes:
  - Primary key on id
  - idx_notifications_user_id

Notification Types:
  - info = Information
  - warning = Warning
  - error = Error

================================================================================
                           USERS AND ROLES
================================================================================

ROLES IN SYSTEM:
1. ADMIN
   - Full system access
   - Can manage all modules
   - Can create staff accounts
   - View audit logs
   - Generate reports

2. STAFF
   - Limited management access
   - Can manage residents
   - Can process bills and payments
   - Cannot manage admin accounts
   - Cannot view audit logs

3. HOMEOWNER / RESIDENT
   - Self-service portal
   - View own bills
   - Make payments
   - View violations
   - Update own profile

================================================================================
                         TABLE RELATIONSHIPS
================================================================================

RESIDENTS (residents)
  - Central entity in the system
  - lot_number uniquely identifies each resident
  - Linked to violations via lot_number
  - Linked to bills via lot_number
  - Linked to payments via lot_number

VIOLATIONS (violations)
  - References residents via lot_number and resident_name

BILLS (bills)
  - References residents via lot_number and resident_name
  - Paid via payments table (bill_reference)

PAYMENTS (payments)
  - References residents via lot_number and resident_name
  - References bills via bill_reference

AUDIT_LOGS (audit_logs)
  - Tracks actions by all user types
  - user_role indicates who performed action

NOTIFICATIONS (notifications)
  - References users via user_id
